P0missing-error-state
LOAD_ERROR silencieux — formulaire vide affiché comme « configuré »
app/(user)/components/settings/alerting/types.ts:70
Contexte
Quand useAlertingPrefs échoue (API down, 401, réseau), le reducer LOAD_ERROR passe loading=false sans changer d'état d'erreur. AlertingSettings.tsx vérifie seulement if(state.loading) — pas d'erreur UI.
Suggestion
Ajouter un champ loadError dans State, le setter dans LOAD_ERROR, et afficher un ErrorBanner inline dans AlertingSettings quand loadError est non-null.
Pourquoi c'est critique
Risque de data loss : l'user croit configurer ses alertes mais écrase potentiellement ses préférences existantes avec un payload vide.
P1missing-loading-state
Dashboard home sans skeleton pendant fetch initial
app/(user)/page.tsx:36
Contexte
refetchState démarre à 'loading' mais aucun skeleton n'est affiché — les KpiCards rendent avec value='—' et les sections Exécution/Radar/Initiatives sont vides.
Suggestion
Skeleton cards animés (animate-pulse) pendant refetchState==='loading' && data===null.
Pourquoi
L'user voit un dashboard vide 3-4s sans savoir si l'app charge ou si aucune donnée n'existe — indiscernable de l'état 'aucune mission'.

hooks/use-conversation-history.ts:83
Contexte
useConversationHistory retourne {loading, error} mais ChatDock.tsx (line 144) appelle le hook sans déstructurer ni error ni loading. Si le fetch échoue, le chat s'ouvre vide sans aucune indication.

L'user ouvre un thread existant, voit le chat vide, ne sait pas si c'est un bug — peut re-taper sa demande inutilement.
