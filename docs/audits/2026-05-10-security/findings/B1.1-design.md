## 1. Schéma DB cible

### Constat actuel

- **`public.users`** (`lib/database.types.ts:3687-3728`) possède déjà une colonne `tenant_ids: string[] | null` (array, nullable).
- **Aucune table `tenants`** n'existe en DB versionnée (grep `CREATE TABLE.*tenants` ne renvoie que `tenant_settings` (`supabase/migrations/0053:10`), `tenant_usage_daily` (`0060`), pas la table parent).
- **`tenant_settings.tenant_id`** est typé `text PRIMARY KEY` (donc string libre — pas UUID).
- 24+ tables ont une colonne `tenant_id` (chat_messages, runs, missions, personas, etc.).
- 2 jobs Inngest dépendent du pattern `users.tenant_ids[0]` pour hydrater `tenantId` et `workspaceId` (`lib/jobs/inngest/functions/monthly-card.ts:73-74`, `lib/jobs/inngest/functions/weekly-digest.ts:86-87`) — donc **`tenantId === workspaceId`** est déjà un invariant implicite dans le code prod.
- L'aggregate analytics (`lib/admin/usage/aggregate.ts:97-100`) lit `users.tenant_ids[0]` comme heuristique fallback.
- **Migration 0051** (`supabase/migrations/0051_runs_tenant_id.sql:22`) backfille `runs.tenant_id` depuis `users.tenant_ids[1]` — confirme `tenant_ids` array existe en DB prod.

### Recommandation : Option C++ (1 user = 1 tenant = 1 workspace, futur collaboratif via membership)

**Choix** : extension pragmatique d'Option C — on garde `users.tenant_ids text[]` (déjà en DB, déjà code-pathé) **mais** on garantit :

- À la création user : un seul tenant_id généré (`uuid_generate_v4()`), inséré comme premier élément de `tenant_ids`.
- Convention : `tenant_ids[0]` = tenant primaire = workspace primaire (`tenantId === workspaceId === tenant_ids[0]`).
- Colonnes nouvelles : `users.primary_tenant_id uuid NOT NULL` (denormalisée — évite `tenant_ids[0]` partout) + `users.primary_workspace_id uuid NOT NULL` (= `primary_tenant_id` à la création, peut diverger plus tard).
- Table parent **`tenants(id uuid PK, owner_user_id uuid, created_at, name text, plan text DEFAULT 'free')`** créée pour gouvernance future (FK source pour audits, billing, plan).

**Justifications** :

| Critère                            | Option A (text PK)                | Option B (N tenants) | Option C++ (1=1 + future membership)                    |
| ---------------------------------- | --------------------------------- | -------------------- | ------------------------------------------------------- |
| Compatible DB existante            | ❌ tenant_settings.tenant_id text | ✓                    | ✓ (garde tenant_ids text[])                             |
| Hearst OS = cockpit perso          | ✓                                 | ❌ overkill          | ✓                                                       |
| Migration users existants          | trivial                           | lourd                | trivial (génère 1 row tenants par user)                 |
| YAGNI futur collaboratif           | ❌ refonte                        | ✓                    | ✓ ajout `tenant_memberships` table sans casser existant |
| Code path existant `tenant_ids[0]` | refacto 5+ files                  | refacto 5+ files     | inchangé (heuristique reste valide)                     |

**Décision** : Option C++ — minimal disruption, maximum backward-compat, futur-proof.

### Schéma cible

```
public.tenants
  id uuid PK default gen_random_uuid()
  owner_user_id uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE
  name text NOT NULL DEFAULT 'Personal Workspace'
  plan text NOT NULL DEFAULT 'free'  -- free | pro | enterprise
  created_at timestamptz NOT NULL DEFAULT now()
  updated_at timestamptz NOT NULL DEFAULT now()

public.users  (modifié)
  -- existing: id, email, role, tenant_ids text[]
  + primary_tenant_id uuid NOT NULL REFERENCES public.tenants(id)
  + primary_workspace_id uuid NOT NULL  -- = primary_tenant_id à la création
```

## 2. Migration SQL — `supabase/migrations/0070_per_user_tenant.sql`

```sql
BEGIN;

-- 1. Table tenants
CREATE TABLE IF NOT EXISTS public.tenants (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_user_id uuid NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,
  name text NOT NULL DEFAULT 'Personal Workspace',
  plan text NOT NULL DEFAULT 'free',
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_tenants_owner ON public.tenants(owner_user_id);

-- 2. Colonnes users (nullable temp pour backfill)
ALTER TABLE public.users
  ADD COLUMN IF NOT EXISTS primary_tenant_id uuid REFERENCES public.tenants(id),
  ADD COLUMN IF NOT EXISTS primary_workspace_id uuid;

-- 3. Backfill : 1 tenant par user existant
DO $$
DECLARE
  u record;
  new_tenant_id uuid;
BEGIN
  FOR u IN SELECT id, email, tenant_ids FROM public.users WHERE primary_tenant_id IS NULL LOOP
    IF u.tenant_ids IS NOT NULL
       AND array_length(u.tenant_ids, 1) >= 1
       AND u.tenant_ids[1] ~ '^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$'
    THEN
      new_tenant_id := u.tenant_ids[1]::uuid;
    ELSE
      new_tenant_id := gen_random_uuid();
    END IF;

    INSERT INTO public.tenants (id, owner_user_id, name)
      VALUES (new_tenant_id, u.id, COALESCE(u.email, 'Personal Workspace'))
      ON CONFLICT (id) DO NOTHING;

    UPDATE public.users
       SET primary_tenant_id = new_tenant_id,
           primary_workspace_id = new_tenant_id,
           tenant_ids = ARRAY[new_tenant_id::text]
     WHERE id = u.id;
  END LOOP;
END $$;

-- 4. Garde-fou : aucun user sans tenant
DO $$
DECLARE c int;
BEGIN
  SELECT count(*) INTO c FROM public.users WHERE primary_tenant_id IS NULL;
  IF c > 0 THEN
    RAISE EXCEPTION 'Migration 0070: % user(s) sans primary_tenant_id après backfill', c;
  END IF;
END $$;

-- 5. NOT NULL strict
ALTER TABLE public.users
  ALTER COLUMN primary_tenant_id SET NOT NULL,
  ALTER COLUMN primary_workspace_id SET NOT NULL;

-- 6. Index pour lookups depuis JWT callback
CREATE INDEX IF NOT EXISTS idx_users_primary_tenant ON public.users(primary_tenant_id);
CREATE INDEX IF NOT EXISTS idx_users_email_lower ON public.users(lower(email));

COMMIT;
```

## 5. SignIn allowlist — F-096

### Recommandation : Stratégie A (domaine email allowlist via env)

**Justification MVP** : Hearst OS cible Adrien + équipe Hearst Corporation + bêta-testeurs connus. Allowlist domaine env-driven est :

- Déployable en 30 min (juste un callback `signIn`)
- Auditable trivialement
- Pas de table DB supplémentaire
- Rollback env-only

Strat B (table `tenant_invites`) reste désirable post-MVP, à introduire en B1.5+ ou B2.

### Implémentation

```ts
// callbacks (avant jwt)
async signIn({ user, account, profile }) {
  if (account?.provider === "dev-bypass") return true;

  const email = profile?.email ?? user?.email;
  if (!email) return false;

  const allowed = (process.env.HEARST_ALLOWED_EMAIL_DOMAINS ?? "")
    .split(",").map((d) => d.trim().toLowerCase()).filter(Boolean);

  if (process.env.NODE_ENV === "production" && allowed.length === 0) {
    console.error("[Auth] HEARST_ALLOWED_EMAIL_DOMAINS not set in prod");
    return false;
  }
  if (allowed.length === 0) return true; // dev only

  const domain = email.split("@")[1]?.toLowerCase() ?? "";
  if (!allowed.includes(domain)) {
    console.warn(`[Auth] signIn rejected: ${domain} not in allowlist`);
    return false;
  }
  return true;
},
```

### Garde env supplémentaire — `lib/env.server.ts`

```ts
if (isProd && !process.env.HEARST_ALLOWED_EMAIL_DOMAINS) {
  throw new Error("[ENV ERROR] HEARST_ALLOWED_EMAIL_DOMAINS required in production");
}
```

Et **retirer** les checks `HEARST_TENANT_ID`/`HEARST_WORKSPACE_ID` env devenus obsolètes (lignes 27-38).
