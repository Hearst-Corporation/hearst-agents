retour à un layout chat-first pur

Hearst OS n'est pas un SaaS — c'est un orchestrateur. Le user parle à l'agent, l'agent agit et ouvre des focals. Le RightPanel (PulseStrip + FocalCard + 4 tuiles + ContentView) est supprimé. Le LeftPanel reste la mémoire des threads. Le body central devient la surface chat-first canonique : conversation + focal au-dessus quand l'agent ouvre un objet.

Principe n°1 · chat-first préservé

Pas de menu nav. Le user parle.

Le LeftPanel reste exactement ce que dit le README : mémoire des threads uniquement. Aucune section Cockpit / Missions / Agents / Assets / KG / Apps. Si Adrien veut voir ses missions, il demande à l'agent ou tape ⌘ K "missions". Pas d'icônes à scanner, pas de page admin déguisée.

  • Refusé : nav SaaS classique en colonne gauche
  • Conservé : threads (Aujourd'hui / Cette semaine / Plus anciennes)
  • Conservé : avatar + settings + sign out en footer
  • Toujours dispo : ⌘ K Commandeur pour atteindre tout
Principe n°2 · 3 états du body

Idle · Conversation · Focal

Le body central n'est pas un dashboard, c'est une surface chat-first qui prend trois formes selon ce que l'agent fait :

  • Idle — pas de thread actif, greeting + ChatInput grand au centre + 4 suggestions contextuelles
  • Conversation — ChatMessages au centre, ChatInput fixe en bas, inline-cards et approvals injectés par l'agent
  • Focal — l'agent ouvre un objet (mission, asset, doc, KG), il occupe le body, ChatInput reste sous le focal pour poursuivre la conversation contextuelle
Principe n°3 · état global → PulseBar

Le moteur reste visible en haut

Ce qui méritait de rester du RightPanel (état moteur, missions actives, à valider, notifs) remonte dans la PulseBar top. Compact, monoligne, scanable sans détourner du chat.

  • Dot teal live + label "Moteur · prêt"
  • Compteurs inline : missions · à valider · connexions
  • Cmd+K central · Voix · Notifs · Focus mode
  • Cliquer sur "à valider" ouvre le focal validation, pas un panel droit
Principe n°4 · approvals dans le chat

Pas de FocalCard de notifs

Les approvals (validation HITL crypto Phase 3 du Battle Plan) ne s'affichent plus dans un panneau droit. Elles arrivent dans la conversation au moment où l'agent les requiert, avec un encart gold-soft inline. Le user lit, décide, l'agent enchaîne. C'est cohérent avec l'agent orchestrateur.

Idem pour les notifs externes (email, message) : l'agent les évoque s'il y a action à prendre, sinon elles attendent dans /notifications accessible via ⌘ K ou la PulseBar.

Principe n°5 · focal absorbe

Plus de 12 ContextRail polymorphes

Les 12 variantes ContextRailFor* (Mission / Asset / Meeting / Voice / KG / Admin / etc.) sont supprimées. Quand l'agent ouvre un focal (mission par exemple), tout le contexte utile est dans le focal : header crumb + actions + body lisible (doc, canvas KG, transcript, etc.) + ChatInput pour itérer.

Les métadonnées techniques (statut, agent, tokens, sources) restent accessibles via le bouton "Journal" du header focal — pas un rail permanent qui prend 22vw.

Impact ADD

3 specs verrouillées à update

Cette refacto reste alignée avec la philosophie chat-first du README (Architecture UX section). Elle touche :

  • F-02 cockpit — supprime la structure « 4 strates RightPanel » au profit de l'état Idle du body central
  • F-18 context-rail — déprécation totale (12 variantes supprimées)
  • F-17 timeline-rail — confirme officiellement "threads only" (renforce le pivot 2026-05-10, ne l'annule pas comme dans la v1 erronée)

Bump v1.1 sur chaque, validation Adrien, puis npm run features:manifest. F-19 stage reste plus ou moins inchangé : le dispatcher Stage devient le routeur Idle/Conv/Focal.
